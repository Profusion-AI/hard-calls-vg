# Integration of Hard Calls Voice Gateway with Remix (Bolt.new) App – Feasibility & Architecture Assessment

## 1. Overview of the Voice Gateway Architecture

The Hard Calls Voice Gateway (VG) is designed as a minimal audio relay service connecting Twilio’s telephony streams to ElevenLabs’ Conversational AI. The open-source repo (Profusion-AI/hard-calls-vg) reveals a lean Node.js service (built with an Express-based Remix server) that primarily handles WebSocket streaming. The core components include:

- **server.js (Express Server & WebSocket Handler):** Initializes an Express HTTP server (or Remix app) and upgrades a specific route (e.g. `/ws/call`) to a WebSocket. This route accepts incoming Twilio Media Stream connections and pairs each with an outgoing ElevenLabs API WebSocket. The code follows a thin relay pattern – on each incoming media message from Twilio (containing a chunk of μ-law audio), the VG forwards it to ElevenLabs; when ElevenLabs returns audio (cloned voice TTS of the AI’s reply), the VG sends it back to Twilio in real-time. The VG does not perform heavy processing like transcription or TTS itself – those tasks are offloaded to ElevenLabs’ API.
- **Project Structure & Modules:** According to documentation, the VG is structured as a Remix app with custom service modules. For example, `app/routes/ws.call.tsx` would handle the WebSocket upgrade, and the logic is implemented in service classes under `services/voice-gateway/` (e.g. `websocket-server.ts`, `twilio-handler.ts`, `elevenlabs-client.ts`, `session-manager.ts`). This separation makes it modular:
  - **Twilio Handler:** Parses incoming media frames (Twilio sends 8kHz μ-law audio frames base64-encoded in JSON messages) and manages the call lifecycle (e.g. Start, Media, Stop events).
  - **ElevenLabs Client:** Maintains a WebSocket to ElevenLabs Conversational AI 2.0, sending user audio and receiving AI-generated audio events.
  - **Session Manager:** Maps the Twilio stream to the ElevenLabs stream, ensuring the correct forwarding of audio in both directions.
- **CLAUDE.md Highlights:** The development guide emphasizes the MVP’s focus on real-time AI call delegation and outlines the target architecture. A mermaid diagram (text form) shows the flow: Browser UI -> triggers call via Backend API -> which signals the Voice Gateway -> that dials out via Twilio and shuttles audio between Twilio and ElevenLabs, returning AI speech to the callee. Key decisions include focusing on outbound calls (no inbound for MVP) and using ElevenLabs’ all-in-one conversation API for STT + LLM + TTS to minimize custom logic. Importantly, the VG leverages native μ-law audio end-to-end to avoid any transcoding overhead – Twilio delivers audio in μ-law 8kHz, and ElevenLabs’ API can accept/provide the same, so the gateway just relays bytes. This keeps latency low and the implementation simple.

In summary, the current VG is a standalone Node service that maintains persistent WebSocket connections and real-time audio forwarding. It is purposely stateless beyond each call session (no database writes in the relay path), and acts as a high-performance conduit between Twilio and the AI. This aligns well with the “thin relay layer with minimal processing overhead” design philosophy. Next, we assess how this architecture fits with the Remix + Netlify front-end scaffold.

## 2. Alignment with Bolt.new (Remix + Netlify) Constraints

Bolt.new provides a Remix+Express application scaffold deployed via Stackblitz to Netlify. This environment imposes significant constraints: Netlify’s deployments run on serverless functions, which are stateless and short-lived. Crucially, long-lived WebSocket connections are not supported on Netlify’s backend. Any attempt to run a persistent WebSocket server (e.g. using socket.io or ws in an Express route) would fail, since Netlify doesn’t allow continuous server processes – functions are cold-started per request and terminate quickly after responding. Similarly, heavy CPU or GPU-bound tasks cannot run on Netlify Lambdas beyond their execution time limits.

How the VG design aligns:

- **Separate Deployment for VG:** The project already plans to deploy the Voice Gateway on Fly.io (or a similar host) separately from the Netlify app. This separation is essential: it offloads all persistent connections (Twilio and ElevenLabs streams) and intensive real-time processing to a service outside Netlify’s serverless platform. By keeping the VG on Fly.io, we avoid Netlify’s WebSocket limitation entirely. The front-end Remix app will interact with the VG over standard HTTP requests or client-initiated WebSockets, which Netlify can handle on the client side. (It’s fine for the browser to maintain a WebSocket to the VG; the limitation is only on Netlify’s server side.)
- **Lightweight Remix Backend:** The Remix+Express backend on Netlify should remain extremely lightweight – essentially functioning as API endpoints that do quick tasks (auth checks, calling external APIs, database ops) and return. This aligns with the VG architecture: the Remix app doesn’t need to run any long-lived processes. For example, initiating a call will likely be a short HTTP request from Remix to the VG or Twilio API (a sub-second operation), well within Lambda limits. All ongoing call audio streaming happens in the VG, not in the Netlify function. This means we’re compliant with Netlify’s stateless lambda model.
- **No GPU/Heavy Compute in Remix:** Any heavy lifting (like audio encoding/decoding, if any, or possibly voice cloning in the future) will occur in the VG or other services. The Remix app will primarily coordinate and use external services (ElevenLabs, Twilio, Supabase). This keeps the Netlify deployment fast and within resource limits.
- **Monorepo vs. Separation Consideration:** Initially, the plan (when considering Vercel/Next.js) was to keep everything in a monorepo and even handle WebSockets in-process with the web app, extracting the VG only if platform limits forced it. Netlify’s restrictions unequivocally force a separation – which we have embraced by using Fly.io for the VG. There is a slight loss of simplicity (now two deployments to manage instead of one monolithic app), but it’s necessary. One upside is that this separation enforces clear module boundaries: the Remix app handles web interactions and short API calls, and the VG handles streaming interactions. We will need to maintain a well-defined interface between them (discussed in the next section). This two-component architecture is still aligned with the MVP goal of lean infrastructure, since each piece is focused and relatively simple.

Potential points of friction or risk:

- **Integration Complexity:** Combining a Netlify-hosted frontend with a Fly.io-hosted VG requires careful integration. CORS and domain issues must be managed (the VG will run on a different domain e.g. `wss://voice.dev.hardcalls.app`). We must ensure secure communication (SSL/WSS) and authentication between the two. Netlify’s stateless nature means the front-end can’t push data to the VG except via new requests – but that’s acceptable since the VG provides its own persistent connection for the call.
- **Latency:** Netlify’s global CDN and Fly.io’s hosting region need consideration. The user’s browser will communicate with Netlify’s edge for initial requests, and the VG likely runs in a specific region (we should deploy it near our Twilio region for low audio latency). The additional round-trip for the browser’s API call (Remix → VG → Twilio) is minor (a few tens of milliseconds) compared to call audio latency, but it should be monitored. The real-time audio path (Twilio <-> VG <-> ElevenLabs) does not involve Netlify at all, which is ideal for performance. The VG aims to add <150ms round-trip latency, which seems achievable given it avoids costly transcoding and runs close to the telephony and AI endpoints.
- **Statefulness:** Netlify’s functions can’t hold session state in memory between calls. This means any state about an ongoing call can’t live in the Remix backend process. We must store needed state externally (database or in the VG) or pass it along with each request. The VG itself will maintain in-memory state for an ongoing call (e.g. the WebSocket pair and buffers), but that is short-term state confined to the call’s lifespan on the VG server.

Overall, the VG’s architecture is compatible with Bolt.new’s Netlify deployment, provided we strictly segregate responsibilities. The key is to treat the Remix app as a stateless orchestrator, and let the VG handle anything requiring persistence or long-running connections. In practice, this means slightly more integration effort (secure API design, session coordination), but it keeps us within all platform limits. The next section outlines how to implement this integration, especially with regard to session management and middleware.

## 3. Middleware and Session State Integration Strategy

To integrate the Remix front-end with the Voice Gateway seamlessly, we need a clear strategy for propagating context and maintaining session state across the two services. The goal is to keep the system lean – avoiding duplication of logic or unauthorized bypass of core app functionality – while respecting that Netlify cannot hold state in memory.

**Front-end as Middleware Orchestrator:** We will use the Remix (Express) server as a middleware layer between the browser UI and the Voice Gateway. The browser should never directly initiate complex operations against Twilio or ElevenLabs; instead, it will call our Remix app, which in turn triggers actions on the VG (or other APIs). This ensures that core functionality stays in-app (the Remix backend is aware of and logging all key events), even though the heavy work happens externally. It also allows us to enforce authentication and business rules in one place.

Concretely:

- **Call Initiation Flow:** When a user initiates a call from the UI (providing the target phone number and perhaps an objective or script for the AI), the browser will send a request to a Remix backend API route (e.g. `POST /api/call/start`). The Remix handler will do the following in a stateless fashion:
  1. **Authenticate & Authorize:** Ensure the user is logged in (via Supabase Auth token or session) and perhaps check they have a cloned voice ready, credits, etc. This is core logic that remains in-app.
  2. **Fetch User Data:** Gather any needed session info, e.g. the user’s ElevenLabs voice ID (which might be stored in Supabase from a prior onboarding step), or any preferences (like language, etc.). Because we can’t persist state in memory, this data should come from the DB or the request payload. For MVP, minimal data (voice ID and call objective text) is likely passed.
  3. **Trigger Voice Gateway:** The Remix function will then call an API endpoint on the VG service to initiate the call. This could be a simple HTTP request like `POST https://voice.dev.hardcalls.app/call` with a JSON payload containing: the `phoneNumber` to dial, the `userVoiceId` or voice settings, the user’s call objective/context, and perhaps a temporary auth token or API key so the VG trusts this request. The VG’s `/call` endpoint (or it could be integrated into the WebSocket handshake) will use this data to start the call.
  4. **Return Response to UI:** Once the VG acknowledges the call start (e.g. returns a 200 OK with a `callId` or Twilio call SID), the Remix function responds to the browser confirming the call is in progress. The UI can then transition to a “calling…” state.
- **Session Tracking:** Since the Remix app can’t hold a live session in memory, we’ll use a combination of client-side state and database:
  - The UI (browser) can keep track that a call has been initiated. If needed, it can poll an API or listen for events to update status.
  - We will maintain a Call Session record in Supabase (or in-memory on the VG, with callbacks to update the DB). For MVP, a simple approach is to write a row to a “calls” table when a call is initiated (with user ID, call SID, status “initiated”). The VG knows the Twilio Call SID as soon as it starts the call; it can include that in its response or send it in a follow-up.
  - As the call progresses, we rely on Twilio’s webhooks or VG’s callbacks to update this record (e.g. “answered”, “completed”, or error codes). Because the VG is stateless across calls, it won’t persist these itself – but it can hit a Remix endpoint to log events, or Twilio can directly hit a Netlify function for call status changes (since Twilio can send HTTP webhooks on call completion, etc.).
- **Ensuring In-App Core Functionality:** By funneling call start through the Remix API, we don’t bypass the main app. This addresses Bolt’s requirement that the application’s core logic reside in-app, not hidden entirely in an external service. The VG is essentially a utility service. The decision to start a call, which user/voice to use, and how to handle errors is coordinated by the Remix app. For example, if the user has no voice cloned yet or is not authorized, the Remix layer will catch that and refuse the call before the VG is ever invoked.
- **State in the VG:** The Voice Gateway will manage real-time state only. When it receives a `/call` request from Remix, it can store the relevant info in memory (e.g. create an object with `callSid`, Twilio WS connection = null initially, ElevenLabs WS = null initially, plus user’s voice ID and prompt). It then calls Twilio’s API to dial the number and attach the media stream. Once Twilio connects to the VG’s `/ws/call` socket (with a stream ID), the VG can match it to the call data (Twilio provides a handshake message with the streamSid and call metadata). The VG then opens a connection to ElevenLabs, sends the conversation initialization (which includes the user’s prompt/objective as `conversation_initiation_client_data` – basically telling the AI what the call is about), and starts forwarding audio. All this state lives in the VG process and is identified primarily by Twilio’s streamSid or call SID. No persistent DB needed in VG – if the process restarts, it loses ongoing calls (which is acceptable; Twilio would in that case terminate those calls anyway).
- **API Design and Security:** The communication between Remix and the VG must be secure:
  - We will deploy the VG behind SSL (https:// and wss:// endpoints).
  - For the VG’s `/call` initiation API, use an authentication mechanism – e.g., a pre-shared secret or an API key in an Authorization header that only the Remix backend knows. This prevents outsiders from hitting the VG to start rogue calls. Netlify functions can securely store this API key (as an env variable) and use it in the request. Alternatively, we could have the user’s Supabase JWT passed and let the VG validate it via Supabase, but that’s overkill; simpler is a shared service token.
  - The WebSocket endpoint `/ws/call` will be invoked by Twilio, not by end-users, so it can be protected by a Twilio-signed JWT or simply by the obscurity of the URL with a secret parameter. Twilio’s `<Stream>` allows specifying `ws://voice.gateway?token=XYZ` that Twilio will include on connect. The VG can verify that token to ensure the stream is authorized. (This is an extra hardening step; Twilio streams can also be IP-locked or protected via their config.)
- **Frontend Listening/Updates:** In the MVP scenario, the user is not actively speaking on the call, so they might not need a live audio feed. But they will want to know the outcome (did the call connect? is it ongoing? completed?). Since Netlify can’t push real-time updates easily, we have a couple of choices:
  - **Polling:** The UI can poll a Remix endpoint (or Supabase) for call status every few seconds. When the status changes to “completed” or “failed”, it can display results (e.g. “Call ended, outcome: success” or show a transcript if available).
  - **Realtime channels:** If we use Supabase, we could leverage its realtime subscription to the calls table. The browser could subscribe (via WebSocket provided by Supabase) to changes in that call record, and get an event when status updates. This is an elegant solution to push status without building our own WebSocket server on Netlify. It keeps within the Jamstack philosophy – Supabase handles the realtime aspect.
  - **Direct WS to VG (optional):** We could consider letting the browser connect directly to the VG’s WebSocket to monitor the call (the VG could broadcast transcript events or status). However, that introduces complexity: the VG would need to handle potentially an extra client WebSocket per call, and filter what it sends (e.g. we might not want to stream raw audio to the UI, but perhaps transcripts of conversation if we have them). For MVP, this is likely not needed. Simpler is to rely on Twilio’s and ElevenLabs’ own processing and only notify when done.
- **Session ID synchronization:** A robust strategy is to use the Twilio Call SID (a unique ID for the call) as the primary key across systems:
  - When the Remix app requests the VG to start a call, the VG (after calling Twilio) will know the Call SID (the Twilio API returns it). The VG can immediately respond to Remix with that SID.
  - The Remix app can store that in the DB (calls table) and use it to track the call.
  - Twilio’s webhooks (status callbacks) will also reference the Call SID, so when a “call completed” webhook hits the Remix (Netlify function), we can match it to the record and update status.
  - The VG’s WebSocket stream uses a Stream SID which is different, but Twilio’s initial message maps Stream SID to Call SID, and the VG can maintain that mapping internally. The Stream SID is only needed for sending audio back on that WebSocket (VG must include streamSid in any “media” messages it sends to Twilio).
- **Stateless Error Handling:** If something goes wrong starting the call (e.g. Twilio API error, or VG is down), the VG’s API should return an error which the Remix function passes back to the UI. The UI can then present the error (e.g. “Call failed to start, please try again”). All of this happens in one request-response, so it’s straightforward.

By following this strategy, we ensure the session state is shared deliberately and not implicitly through server memory. The Remix app acts as a gatekeeper and tracker (with DB support), and the VG acts as the session host for the live call. This keeps the integration lean: the VG doesn’t need to know about the user beyond what’s needed for the call (voice ID and prompt), and the Remix app doesn’t manage any streaming directly. It’s a clean producer/consumer split with clearly defined handoff points.

## 4. Risks of Divergent Session State and Fallback Plans

With a distributed architecture, there are risks that the front-end’s view of the call and the VG’s reality might fall out of sync. Let’s identify potential failure scenarios and how to mitigate them:

- **Call Initiation Failures:** After the user presses “Call”, any number of things could fail – the VG might be unreachable, Twilio could reject the call (invalid number, etc.), or ElevenLabs might not connect. If the VG’s `/call` API call fails or times out, the Remix backend will receive an error and can relay that to the UI. The user should get immediate feedback (“Unable to start call: service unavailable” or similar). **Fallback:** Ensure the Remix API call to VG has a reasonable timeout, and handle non-200 responses. Log the error (to Supabase or logging service) for debugging. The user can then retry. This prevents a case where the UI thinks a call is happening when it never did.
- **Mid-Call VG Failure:** If the Voice Gateway crashes or restarts mid-call (or the Fly.io instance is briefly down), the WebSocket connection from Twilio will drop. Twilio’s Media Stream will detect the socket closure; by default, Twilio will terminate the call when the media stream disconnects unexpectedly (since our AI is essentially the “caller’s voice”). In other words, the callee will likely get silence and then the call will hang up. From the system’s perspective, Twilio will fire a “call disconnected” webhook. **Mitigation:** We should configure a Twilio Status Callback URL for the call (Twilio allows specifying a webhook for events like call completed, error, etc.). This callback can point to a Remix function (Netlify) that updates the call record to “failed” or “completed”. That way, even if the VG dies, the front-end will eventually learn the call ended (via the DB or polling). To be robust, the VG could also attempt to reconnect or handle partial failures (but given the time constraints of MVP, automatic failover is probably out of scope). Our fallback is simply to ensure we signal the end state properly. Users might have to reinitiate the call if it dropped. In production, we’d consider running multiple VG instances for redundancy (horizontal scaling), but MVP can start with one and accept the risk of downtime.
- **ElevenLabs API Errors or Latency:** If the ElevenLabs Conversational AI encounters an error (e.g., it can’t process audio or yields no response), the VG should handle this gracefully. For instance, if no audio response is received for a certain duration, the VG could send a “clear” message to Twilio (to flush any buffered silence) and possibly inject a TTS message like “<sorry, something went wrong>” if we have a backup TTS. However, since we rely on ElevenLabs for TTS, a failure there means we have no AI voice to continue. **Fallback Plan:** Define timeouts – e.g., if no response from ElevenLabs in, say, 5 seconds, the VG can terminate the call. Twilio will then hang up on the callee. We should document this as a known limitation for MVP (calls might drop on AI failure). In the future, we could implement a retry or use a secondary TTS engine (like Chatterbox, see next section) as a backup for a final apology message.
- **Session State Divergence:** The front-end might think the call is ongoing when it’s not, or vice versa. This could happen if callbacks are delayed or lost. **Solution:** Use multiple signals to determine state:
  - When the VG finishes handling a call (either normally or due to error), it can send an HTTP request to a Remix endpoint (e.g., `/api/call/end`) with the call SID and reason. This duplicates what Twilio’s callback does, but provides redundancy. The Remix function can then update the DB status.
  - The UI, on its side, could have a maximum call duration timer – if a call exceeds, say, 5 minutes (or some expected length) without a completion signal, it queries the status again or assumes it ended. This is a safety net against any missed webhook.
  - Additionally, Twilio’s “mark” events can help synchronize audio play-out. The VG can use Mark messages to know when Twilio has finished playing the last chunk of AI audio. This is more for synchronizing turn-taking than session state, but it helps ensure the conversation flows properly. If divergence occurred (e.g., Twilio still waiting to play audio but VG thinks it’s done), a Mark event would clarify.
- **User Cancellation:** MVP doesn’t explicitly mention the user canceling a call mid-way (since the AI is handling it autonomously). But if we did need to allow the user to abort, the front-end would call a Remix endpoint which in turn would instruct the VG to hang up (either by calling Twilio’s REST API to update the call and hang up, or by sending a WebSocket “clear” and a special signal to ElevenLabs to stop). This scenario is mostly about keeping state in sync: if user presses “End Call”, we must update both Twilio and VG. This again underscores that Twilio’s call SID should be the single source of truth – ending the call via Twilio ensures both sides drop.

In summary, to handle these risks we will implement redundant state reporting and timeouts:

- Use Twilio’s reliable callbacks to update call status in the Remix app’s database.
- Have the VG also report end-of-call, to cover cases where Twilio’s callback might not fire (for example, if the call never fully connected).
- Make the front-end resilient by polling or subscribing to call status, and by handling error responses on initiation.
- Keep user informed – if something fails, surface it (don’t just silently end). For MVP, a simple alert or status message suffices (“Call ended due to an error.”).

By planning for these edge cases now, we can ensure a more polished MVP where “hard calls” don’t fail in hard-to-understand ways. Even if a failure occurs, the system will fail gracefully (cleanly ending the call and informing the user) rather than hanging indefinitely or requiring a manual refresh.

## 5. Incorporating Chatterbox for Voice Cloning in MVP

The product requires demonstrating integration with Chatterbox (Resemble AI’s open-source voice cloning TTS) alongside ElevenLabs. Chatterbox is an exciting alternative – it’s a local TTS model boasting quality on par with ElevenLabs – but incorporating it into a real-time call pipeline introduces challenges. Here’s how we can approach it:

- **Role of Chatterbox:** Unlike ElevenLabs, which provides a full STT→LLM→TTS pipeline via API, Chatterbox is primarily a text-to-speech voice cloning model. It can generate speech in a cloned voice given some reference audio and text. It does not do speech recognition or dialogue management. This means if we want to use Chatterbox, we would use it in place of ElevenLabs’ TTS component, not the entire conversation. In practice, we might still use ElevenLabs (or another service) for STT and possibly the LLM response, then pipe the text into Chatterbox for final voice rendering.
- **MVP Integration Strategy:** Given the tight timeline (MVP by June 20), a full replacement of ElevenLabs with a custom pipeline is risky. Instead, we can demonstrate Chatterbox on a smaller scale to fulfill the requirement:
  - **Voice Sample Upload & Clone:** When a user provides a voice sample (as part of onboarding), we primarily send it to ElevenLabs to create a voice profile (for use in calls). In parallel, we can use that same sample to run through Chatterbox’s process. For example, Chatterbox’s Hugging Face demo allows providing a reference audio and text to synthesize voice. We could automate this by calling an API (perhaps using Resemble’s hosted version or a local container).
  - **Playback Demo:** The app could have a “Preview your cloned voice (Open Source)” button, which uses Chatterbox to speak a preset phrase (like “Hello, I’m your AI assistant”) in the user’s voice. This would be generated either on the server side (if we have Chatterbox running on a server or on the VG machine with a lightweight model) and then streamed to the UI as an audio file, or on the client side if feasible (likely server side is better due to performance).
  - **Real-Time Calls (Optional for MVP):** If we ambitiously try to use Chatterbox in an actual call, the architecture would change: The VG would receive text responses (meaning we’d have to handle STT and LLM ourselves). Given that’s not in the current scope (we rely on ElevenLabs for those), doing this fully by MVP is probably not feasible. Instead, we might simulate one turn with Chatterbox for demo purposes. For instance, for a scripted demo call, we could generate some responses with Chatterbox beforehand. However, this is not truly real-time – Chatterbox is reportedly fast (~200ms latency for generation), but that likely assumes a good GPU and a small text snippet. Running it on a CPU or large text could be slower.
- **Integration Points:** To include Chatterbox properly in the architecture, we should design the VG or a related service to support pluggable TTS engines:
  - We could abstract the “TTS output” in the VG’s code. Currently the VG just relays ElevenLabs audio. We could modify it such that if a certain flag is set (e.g. `useChatterbox=true` for a call or for all calls), the VG will capture the transcribed text + AI response text instead of audio. In ElevenLabs’ streaming API, we might get intermediate transcripts and a final response text (especially if using their API in a different mode). If not, an alternative is to use a different LLM (OpenAI or Claude) to generate the AI’s text reply given the user’s transcribed input, then synthesize that via Chatterbox. This essentially means building a parallel pipeline:
    - Twilio audio -> (maybe use ElevenLabs STT only, or use a generic STT) -> text -> OpenAI GPT-4 (with the call objective prompt) -> response text -> Chatterbox TTS -> audio -> back to Twilio.
  - Implementing all of that reliably is a heavy lift for MVP. Thus, a more realistic plan is partial integration now, full integration later. For MVP, focus on:
  - **Setting up the infrastructure to run Chatterbox:** possibly a small GPU-enabled instance or using a cloud function that can run the model. ResembleAI open-sourced a 0.5B model; this might fit on CPU with slow speed, but for 200ms latency as advertised, a GPU is needed. Fly.io doesn’t (currently) offer GPUs, so maybe we use a separate service or a local machine for demo. Another idea is using Hugging Face’s API for the model if available.
  - **Validate that we can clone the user’s voice using Chatterbox by generating a sample phrase.** This will prove the concept to stakeholders (a “wow” factor: we’re not tied to a closed provider; we can do voice cloning ourselves).
  - **Hide the complexity behind a feature toggle.** For example, in the UI or config, have an option “Use local TTS”. Initially, this might not be exposed to end users, but we can test it internally.
  - **Timeline Consideration:** We should budget a bit of time in the second week of development to integrate Chatterbox at least in a basic way. The PRD likely calls for demonstrating voice cloning – that implies at minimum playing back the user’s cloned voice. It might be acceptable if the call itself still uses ElevenLabs, as long as we show Chatterbox working in the app somewhere. For instance, after the call, we could say “We used ElevenLabs for the call, but listen to this message generated with our open-source model using your voice!” – and play an audio clip. That satisfies the demonstration requirement without risking the core call functionality.
- **Future Integration:** Once MVP is delivered, we can work on deeper integration: possibly replacing ElevenLabs TTS with Chatterbox completely to reduce cost. We’d then likely run our own speech pipeline. That will require more development (speech recognition engine, using an LLM for dialogue – perhaps OpenAI or an on-prem model – and tying into Chatterbox for output). The VG is already structured to allow this swap, since it’s basically a relay. We can eventually add a branch: if “engine = ElevenLabs” use the Eleven WS, if “engine = Local” use local STT + LLM + TTS. The architecture is modular enough to accommodate this later, as long as we design with that in mind (e.g. keep the ElevenLabs integration code separate and abstract the interface to “AI Engine”).

In summary, our recommendation for MVP is to incorporate Chatterbox in a limited but impactful way:

- Use it to generate a cloned voice sample in-app (e.g. a preview or a post-call summary in user’s voice).
- Prepare the code structure to allow swapping the TTS engine, but do not risk the live call flow with untested pipelines.
- Highlight this capability in the demo to show we can pivot to open-source voice cloning if needed (avoiding vendor lock-in), which addresses a key concern.
- On a technical level, ensure the audio format from Chatterbox is μ-law 8kHz or convert it, so if we ever feed it into Twilio, it’s compatible (Chatterbox likely produces 22kHz PCM by default; we’d need to downsample and μ-law encode – a task for the VG or a preprocessing step).

Finally, note that running Chatterbox will require more compute – possibly a GPU service. This is outside Netlify’s scope (can’t run ML on Lambda), and likely we wouldn’t run it on the same VG instance handling live calls (to avoid blocking the event loop). If we integrate it server-side, we may spawn a child process or microservice for it. This is another reason to limit usage in MVP (maybe generate once and cache it). These details can be managed as we integrate, ensuring the MVP remains stable.

## 6. Dividing Responsibilities: What to Offload to Remix vs. VG

To preserve modularity and keep each part of the system efficient, we delineate responsibilities clearly between the Remix app (Netlify) and the Voice Gateway (Fly.io):

**What the Remix+Express App Handles (Netlify side):**

- **User Authentication & Profiles:** All user signup/login and voice sample upload flows live here. Supabase Auth is integrated on the front-end; the back-end (Remix) verifies JWTs or session on each request. For example, the endpoint to upload a voice sample (to ElevenLabs or store in S3) is in Remix. The front-end also provides the UI for recording and reviewing the sample.
- **API Routes for Business Logic:** Any “decision making” API is on Remix. Starting a call, as discussed, goes through a Remix route. Likewise, if we add features like ending a call, fetching call history, or saving call notes, those are implemented as Remix functions (which may in turn call external services or the VG). This ensures all core user actions funnel through the main app, which can enforce rules and log events.
- **Calling Third-Party APIs (when light):** The Remix app can directly call external APIs for short tasks. For instance:
  - It might call ElevenLabs API to create a voice clone profile (one-time HTTP request when user submits their voice; this is not latency-sensitive to the call).
  - It could call Twilio REST API for non-streaming tasks (e.g., to buy a phone number, or to initiate the call if we choose that route). A Twilio call setup is quick (<1s) so it fits in a lambda.
  - It will call Supabase (PostgreSQL) for any data it needs (user profile, call logs, etc.).
- **Session Persistence & Logging:** The Remix app, being stateless itself, uses the database to persist any long-lived state:
  - Users’ voice IDs from ElevenLabs (and potentially a reference to their Chatterbox voice data if we store something for that).
  - Active call records and past call records (time, participants, outcome, maybe a transcript or summary if we decide to save that).
  - Error logs or metrics can also be stored or sent to a monitoring service from within these functions.
- **Front-End UI State:** While not a back-end responsibility, it’s worth noting the front-end React app (Remix’s client side) will manage local state for the call UI (e.g., showing “connecting…”, “on call”, and final results). It might receive prompts to update state from the back-end via responses or subscriptions.

To summarize, Remix/Netlify is the brains and memory: it knows about users, what calls are being made, and interacts with external APIs for setup and teardown. It remains “lightweight” in that it doesn’t do lengthy processing – any single request is quick – but it’s critical for orchestrating flows.

**What the Voice Gateway Service Handles (Fly.io side):**

- **Real-Time Telephony & Audio Stream Processing:** The VG is the muscle and ears/mouth: it deals with streaming audio in real time. Its tasks:
  - Maintain the WebSocket server endpoint to which Twilio streams the call audio. This means handling the WebSocket upgrade and reading a continuous stream of messages (JSON with base64 audio chunks and event signals). It must parse these and extract the raw audio bytes.
  - Establish and manage the WebSocket client connection to ElevenLabs’ streaming API. The VG sends audio bytes (likely encapsulated in their protocol messages, e.g., a `user_audio_chunk` message) to ElevenLabs, and in return receives AI responses – possibly as audio events (audio data chunks for the TTS) and maybe interim text. The VG streams those out to Twilio as soon as they arrive to achieve low latency (the thin relay forwards data as it comes).
  - Enforce audio format compatibility: Twilio requires outgoing audio in μ-law 8-bit, base64-encoded in a JSON “media” message. If ElevenLabs returns PCM 16-bit, 24kHz audio, the VG would convert it to 8kHz μ-law. However, documentation suggests ElevenLabs can directly stream μ-law/8000 (the VG plan noted native μ-law support, likely meaning we can request μ-law format from ElevenLabs). If true, the VG can forward audio bytes with minimal transformation. If any encoding is needed, it will be done here in the VG (likely using a fast audio library or even Web Audio API if available in Node, or a simple μ-law encoder for raw bytes).
  - Handle Twilio bidirectional messaging: The VG uses Twilio’s Media Stream protocol to send audio back to Twilio. As shown in Twilio’s docs, our server can send JSON messages of type "media" containing base64 μ-law audio payload to play audio to the caller. The VG is responsible for constructing these messages properly with the correct streamSid and ensuring no audio header bytes are included. This is a critical real-time function – timings and order matter, so the VG probably buffers just enough and uses Twilio’s “mark” events to sync if needed.
  - Manage call session lifecycle: The VG must listen for Twilio’s “start” event (which includes stream SID and call SID when the stream begins) and Twilio’s “stop” event (when the call or stream ends). It also monitors ElevenLabs’ connection – if the AI signals the end of conversation or if the socket closes. The VG’s session-manager.ts likely coordinates cleanup: when either side is done, close the other side and free resources. If Twilio sends a mark or clear command (or if we send one), handle accordingly. Essentially, VG is a state machine reacting to events from both sides.
  - Minimal logic on AI content: The VG does not interpret the conversation or perform NLP on the text. It’s agnostic to call content. This keeps it modular – if we later swap out ElevenLabs for another AI service, the VG code changes only in how it connects/formats data, not in the high-level call logic.
  - Twilio API Integration (Dialing out): If we decide the VG should initiate outbound calls, it will contain the Twilio REST API call. That’s a simple HTTP POST with the callee number and a TwiML instruction to start streaming. This is short-lived and fine to do on the VG (the VG has to know Twilio credentials either way for streaming auth). The benefit is that the VG can directly control the call setup and be immediately ready to handle the stream. The downside is the Remix app must wait for the VG to complete this step via its `/call` API. Both approaches (dial from Remix or dial from VG) are viable; we just need to ensure one of them does it. We lean towards doing it in VG for encapsulation, but it’s a minor design choice.
  - No long-term storage or heavy computation: The VG, as planned, does not interface with the database or maintain any data between calls. It shouldn’t, for example, try to update Supabase when a call ends – that would slow it down or complicate error handling if the DB call fails. Instead, it should delegate that to the Remix side via an API callback, as discussed. Also, the VG should not block on any CPU-heavy tasks in the Node event loop; forwarding audio is I/O-bound and should be fine. If we incorporate Chatterbox TTS down the line within the VG, we’d have to spawn a separate process or thread for it to avoid blocking the main loop, or use an async service.
  - Scalability and Multi-Call: For MVP, we might run a single instance of the VG. It’s capable of handling multiple simultaneous WebSocket pairs (Node.js can handle many connections, the bottleneck would be CPU usage for encoding or memory for buffering). We need to ensure the code is written to handle concurrent calls (likely with a map of session objects keyed by stream or call ID). If usage grows, scaling out horizontally on Fly.io by running multiple instances is straightforward, but then we must ensure each new call knows which instance to use (usually a load balancer can route Twilio’s webhook to one of the instances, and since calls are independent, that’s okay). For now, one instance is sufficient; we just note that the architecture doesn’t preclude scaling (stateless nature helps here).

**Preserving Modularity:**

The separation we’ve outlined ensures each side can be worked on and tested somewhat independently. For instance, we can simulate calls to the VG using Twilio’s mock or even send pre-recorded audio to it to test the ElevenLabs integration, without involving the front-end. Conversely, the front-end can be tested with a stub VG (for example, hitting a dummy endpoint that immediately responds) to ensure the UI flow works. The interface between them – primarily the `/call` start API and perhaps a `/call` end callback – are well-defined.

We also achieve a form of loose coupling: the Remix app needs minimal knowledge of how the VG does its job (only that it will initiate a call and stream AI voice), and the VG doesn’t know why a call is being made (it doesn’t care about the user’s goal beyond the prompt it’s given). This modular design means future changes (like integrating a different AI service or moving the telephony to a different provider) would require changes mostly in one component, with the API between them remaining stable or only slightly adjusted.

One area of coupling is the content of the AI prompt: The Remix app might pass an “objective” or some user context that needs to go to ElevenLabs’ conversation API. We must ensure this is conveyed accurately via the VG. Likely, the VG’s ElevenLabs client will accept a field for initial prompt. We should use that so the AI has the user’s intended context (for example, “You are calling the cable company to negotiate a lower bill” as the initial instruction). This is core to the AI’s behavior but is just data flowing through the VG. Modularity is maintained as long as the VG doesn’t need to interpret it, just pass it along.

**Offloading Work Appropriately:**

In summary, our architecture offloads real-time streaming and audio transformation to the VG (because Netlify can’t do it), and keeps session control, user interaction, and persistence in the Remix app (because that’s where our product logic lives and it plays to Netlify’s strengths of short bursts of compute). This division is optimal for the constraints given. It also makes testing easier: e.g., load-testing the VG for audio streaming can be done separately from the front-end.

Finally, by preserving this separation, we maintain the optionality of services – for example, if tomorrow we wanted to support a different front-end channel (say a mobile app or a CLI to initiate calls), it could directly talk to the VG’s API or via the same backend, without redesigning the streaming logic. Similarly, if a new AI voice service comes along, we can integrate it in the VG without changing how the front-end triggers calls.

## 7. Actionable Recommendations and Next Steps

Based on the above analysis, here is a concise list of recommendations and action items as CTO, to ensure the successful integration of the Voice Gateway with the Remix app under Bolt.new/Netlify constraints:

- **Use a Clear API Contract Between Frontend and VG:** Define endpoints on the VG such as `/call/start` (trigger outbound call) and possibly `/call/stop` (end a call, if needed) or `/call/status`. Secure these with a token. Implement the Remix side to call these endpoints when users initiate or end calls. This keeps the front-end simple and the VG fully in charge of call execution.
- **Keep Netlify Functions Stateless and Fast:** Do not attempt any long-lived workflows on Netlify. For example, do not try to proxy the WebSocket through Netlify. All streaming must happen directly between Twilio→VG and VG→ElevenLabs. The Remix functions should stick to under a few seconds execution. This aligns with Netlify’s limitations. If any operation seems to need waiting (e.g. waiting for VG response), convert it to an async callback model instead of holding the function open.
- **Leverage Supabase for Shared State:** For any state that needs to be visible to both the Remix app and potentially the VG, consider using Supabase as a lightweight message bus or state store. For instance, writing a “call started” record that includes the call SID and user ID, and updating it when the call ends. The VG can either update via a secured direct DB call (using Supabase service key) or, safer, via an API call to Remix. This ensures that even though the VG and Remix are separate, they have a common reference of truth for calls. Additionally, the real-time subscriptions can push updates to the UI without custom WebSocket coding on Netlify.
- **Monitor and Optimize Latency in VG:** Since the VG introduces an extra hop, we need to keep it efficient. Ensure that audio forwarding in the VG is done in a streaming fashion (don’t accumulate large chunks; forward bytes as soon as they arrive). Utilize Node.js buffers and binary frames for WebSocket where possible. The design choice to stick with μ-law audio is wise – continue with that to avoid expensive transcoding. This will help us meet the <150ms added latency goal. We should measure round-trip audio latency in testing and, if needed, tune buffer sizes or use lower-level socket techniques. Also deploy the VG to a Fly.io region that is close to Twilio’s media POP and ElevenLabs’ servers (for example, if using Twilio US East, deploy VG in US East). This minimizes network latency.
- **Implement Robust Error Handling in VG:** The VG’s session-manager should handle error events from either side without crashing the process. Use try/catch around WebSocket event handlers. If the ElevenLabs socket errors out, log it and send a “clear” to Twilio (so the caller isn’t stuck hearing nothing). If Twilio disconnects, ensure the ElevenLabs socket is closed promptly. This will maintain stability. Also consider rate-limiting or rejecting new calls if the service is overloaded, instead of letting it degrade ungracefully.
- **Integrate Twilio and ElevenLabs Features Smartly:** Use Twilio’s bidirectional media support fully – e.g., send “mark” events to synchronize (the Twilio docs show how to send a mark and get confirmation when audio has played). This can help if the AI voice is talking too continuously – we can insert marks or use the mark when the AI finishes speaking to signal Twilio to listen again. Also leverage ElevenLabs Conversation API features: provide the conversation context (user’s objective) on startup, and utilize any turn-taking signals or end-of-speech detection they have. The ElevenLabs streaming likely emits an audio_started and audio_finished event – tie those into Twilio marks or just log them for now.
- **Plan a Basic Chatterbox Demo:** Allocate time to set up the Chatterbox TTS model. Perhaps use the HuggingFace space or spin up a Docker with it. Test it with a sample voice (even before integrating user voices). Once confident, integrate a “Voice Preview” feature: after the user records their voice and we clone it on ElevenLabs, call Chatterbox to generate a sample phrase. This can be done in a Netlify function if we call an external service (e.g., HuggingFace Inference API) – that keeps it stateless on our side but uses an external compute. Alternatively, pre-generate a couple of audio clips using Chatterbox offline and store them. The key is to show that this voice sounds as good as ElevenLabs. Action item: prepare a short script for the voice preview (one or two sentences that highlight tone) and get that working with Chatterbox. We will showcase this in the beta launch to underscore our open-source capabilities.
- **Use Feature Flags for Chatterbox vs ElevenLabs:** Wrap the TTS selection in a config flag. By default, use ElevenLabs for reliability. But have a toggle (even if only in code or an admin setting) to route TTS through Chatterbox. This will allow internal testing of full calls with Chatterbox when we’re ready, without disturbing normal users. It also provides a path to gradually roll out the open-source TTS if it becomes viable in production (we could A/B test calls using the flag, etc.).
- **Security and Compliance:** Ensure that any secrets (API keys for Twilio, ElevenLabs, ResembleAI if any) are kept out of client-side code. They should reside in Netlify env vars (for Remix to use on secure functions) or in the Fly.io instance env (for VG to use). The communication channels should all be encrypted (WSS for Twilio<->VG, HTTPS for VG API, etc.). Also, if storing call audio or transcripts, be mindful of privacy – MVP may not store full audio, which is fine. Just log high-level info.
- **Testing and Monitoring:** Before public beta, perform end-to-end tests:
  - Simulate a call with a known script to see if the AI handles it via ElevenLabs. Check the audio quality and latency by having a team member be the callee.
  - Test failure modes (kill the VG mid-call, see if the system recovers and updates status).
  - If possible, load test the VG with 2-3 simultaneous calls to ensure it doesn’t mis-route audio.
  - Monitor resource usage on the VG (Fly.io provides metrics). Ensure CPU is not maxing out with μ-law encoding/decoding – if it is, consider using native addons or a small C++ lib for that conversion.
  - Set up basic monitoring/alerts: if the VG process crashes or if a call fails, make sure it’s logged. Maybe integrate a logging service or at least CloudWatch-style logs on Fly.io and Netlify function logs for debugging issues.

By following these recommendations, we will address the main feasibility concerns and edge cases for the Hard Calls MVP. The Voice Gateway + Remix architecture is viable under the given constraints, as long as we carefully manage the boundary between them. The end result will be an AI-powered calling system where the user’s experience is smooth (thanks to the Remix app handling flow and state) and the real-time performance is robust (thanks to the purpose-built VG service). This modular yet integrated approach positions us well not only to deliver the MVP on time, but also to extend it with new capabilities (like open-source voice cloning, or alternate AI backends) in future iterations, without a rewrite of the whole system.
